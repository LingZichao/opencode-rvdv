# Case 005: 状态与系统寄存器（量化增强版）

请生成一个 FORCE-RISCV ISG 脚本，目标是在 RV64 单线程中构造“系统状态建立 -> 状态变更 -> 读回验证”的可观测 CSR 行为序列。

## 任务目标

通过可写 CSR 的多轮读改写，形成可追踪状态迁移路径，使 trace 中能看到初始值、变更值和验证值三者关系。

## 量化要求

1. CSR 相关指令不少于 14 条。
2. 至少覆盖 3 种 CSR 操作形式（例如 `csrrw` / `csrrs` / `csrrc` 或其立即数变体）。
3. 至少操作 2 个可写 CSR（例如 `mstatus`、`mscratch`、`mtvec`，按当前权限级可写性选择）。
4. 每个目标 CSR 至少经历一次“写入或置位/清位”以及一次“读回验证”。
5. 至少 2 轮状态切换（初始配置 -> 变更配置）。
6. 辅助整数指令可存在，但 CSR 指令占比不低于 25%。
7. 结束前写 x10/a0 = 0，然后发出 M5EXIT##RISCV。

## 指令级结构要求

1. 阶段 A（基线建立）：
	- 读取目标 CSR 当前值作为基线。
	- 使用掩码构造写入值，完成第一次状态设置。
2. 阶段 B（状态变更）：
	- 对同一 CSR 字段进行第二次变更（置位或清位）。
	- 至少一次使用 read-modify-write 风格（如 `csrrs/csrrc`）。
3. 阶段 C（读回验证）：
	- 将 CSR 读回到 GPR。
	- 通过 `andi/xori/sub/sltiu` 等整数比较链检查关键位是否符合预期。
4. 必须显式区分“控制寄存器值载体 GPR”和“比较掩码 GPR”，避免验证过程互相污染。

## 数据编排要求

1. 为每个目标 CSR 定义：
	- baseline_mask（观测位）
	- set_mask（第一轮设置位）
	- toggle_or_clear_mask（第二轮变更位）
2. 验证阶段至少检查两类条件：
	- 被设置位在读回值中为 1
	- 被清除位在读回值中为 0
3. 若 CSR 某些位为 WPRI/WARL，不得将这些位作为硬性判定位，需在说明中声明仅校验可预期位。

## 通过判定

1. trace 中可识别每个目标 CSR 的“读取 -> 修改 -> 读取”闭环。
2. 两轮状态切换后，关键位读回结果符合掩码判定。
3. 不依赖异常路径或特权切换副作用完成验证。
4. 脚本可正常结束，无死循环。

## 实现边界

- 仅限 RV64 单线程。
- 不把分页、异常处理、多核、向量作为主线。
- 重点是 CSR 状态迁移与验证可观测性，不要求覆盖全部特权语义。

## 交付物

- 一个完整的 FORCE-RISCV Python 脚本。
- 一段简短说明，解释目标 CSR 选择、两轮状态切换设计、以及读回判定方式。
