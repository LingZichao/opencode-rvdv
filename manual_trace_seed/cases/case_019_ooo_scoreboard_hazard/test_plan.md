# Case 019: OoO 记分牌与依赖冒险专项（微架构功能）

请生成一个 RV64 单线程 ISG 脚本，目标是在短窗口内集中制造 RAW/WAW/WAR 混合依赖，专项验证 OoO 调度与记分牌冲突处理行为。

## 任务目标

该 case 重点是微架构层面的依赖仲裁与发射阻塞恢复，不是普通算术覆盖。需要能在 trace 中观察到依赖链构造、冲突窗口和恢复后继续发射。

## 量化要求

1. 指令总数 110 到 190 条。
2. 整数 ALU 指令不少于 50 条。
3. 显式依赖对不少于 20 组，其中：
   - RAW 不少于 10 组
   - WAW 不少于 4 组
   - WAR 不少于 4 组
4. 至少 3 条长度 >= 4 的依赖链。
5. 至少 2 个“热点寄存器”被重复用于多类依赖。
6. 至少 2 个冲突窗口后出现恢复块（继续执行并收敛）。
7. 结束前写 x10/a0 = 0，然后发出 M5EXIT##RISCV。

## 指令级结构要求

1. 阶段 A：预热与热点寄存器初始化。
2. 阶段 B：RAW 主链构造（短间隔 writer-reader）。
3. 阶段 C：WAW/WAR 干扰插入（对同一热点寄存器）。
4. 阶段 D：混合冲突窗口与恢复块。
5. 阶段 E：收敛退出。

## 数据编排要求

1. 定义 `HOT_GPR_SET`（至少 2 个）与 `AUX_GPR_SET`（至少 4 个）。
2. 依赖链构造应固定“链头寄存器”，中间插入可控干扰寄存器操作。
3. 至少一组窗口只改变依赖类型（RAW->WAW 或 RAW->WAR），保持其余模板不变。
4. 每个冲突窗口前后写入路径签名寄存器，支持 trace 阶段化。

## 通过判定

1. trace 可统计三类依赖对并区分冲突窗口。
2. 冲突窗口后存在恢复块并继续执行。
3. 脚本最终正常退出。

## 实现边界

- 仅限 RV64 单线程。
- 不引入向量主线。
- 重点是 OoO 依赖仲裁与记分牌冲突处理可观测性。

## 交付物

- 一个完整的 FORCE-RISCV Python 脚本。
- 一段简短说明，解释依赖链、冲突窗口与恢复行为如何满足指标。
