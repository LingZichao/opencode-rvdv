# Case 008: 异常委托与特权往返链路（高难度）

请生成一个 RV64 单线程 ISG 脚本，目标是在同一执行会话中构造“异常源触发 -> 委托目标处理 -> 特权返回 -> 后续验证”的完整往返闭环。

## 任务目标

该 case 重点施压异常路径与特权切换细节，不只触发异常，还要展示处理归属变化和返回后语义连续性。

## 量化要求

1. 脚本总指令数控制在 110 到 190 条。
2. 特权/系统状态相关指令不少于 20 条。
3. 异常相关事件至少 3 次，其中至少覆盖 2 种异常来源（例如 ecall 与访存/取指异常候选）。
4. 至少 2 次异常应发生在不同源特权级上下文中。
5. 异常后返回路径必须至少 2 次成功落回到预期后继块（可通过标记块确认）。
6. 至少 1 次异常后执行读回验证，确认状态没有失控漂移。
7. 结束前写 x10/a0 = 0，然后发出 M5EXIT##RISCV。

## 指令级结构要求

1. 至少分 5 个阶段：
   - 阶段 A：基线建立与异常处理上下文准备
   - 阶段 B：第一类异常触发与处理
   - 阶段 C：第二类异常触发与处理
   - 阶段 D：特权往返后状态一致性验证
   - 阶段 E：收敛退出
2. 阶段 B 与阶段 C 的异常触发机制需有差异，避免重复同一触发模板。
3. 每次异常路径后必须有可识别返回锚点块，证明执行流回到主线。
4. 至少一处返回后继续执行依赖前文状态的操作，验证上下文衔接。

## 数据编排要求

1. 预先定义异常触发点集合 TRAP_SET（至少 3 个），每个触发点关联：
   - 触发前标记寄存器状态
   - 预期目标处理级别
   - 返回后验证寄存器集合
2. 针对每个触发点，保留触发前后最少 3 条指令作为可观测边界。
3. 读回验证应覆盖：
   - 至少一个系统状态值
   - 至少一个普通数据路径寄存器值
4. 任意两次异常触发之间应插入短常规块，避免异常事件连续粘连不可分辨。

## 通过判定

1. trace 中可定位至少 3 次异常事件，且存在至少 2 类触发来源。
2. 每次异常事件都能定位到返回后锚点块，无执行流丢失。
3. 往返后关键状态读回结果满足预期一致性。
4. 脚本最终正常退出。

## 实现边界

- 仅限 RV64 单线程。
- 不以向量和浮点为主线。
- 重点是异常委托、处理和返回语义的可观测性。

## 交付物

- 一个完整的 FORCE-RISCV Python 脚本。
- 一段简短说明，按“触发 -> 处理 -> 返回 -> 验证”的闭环解释如何满足量化指标。
