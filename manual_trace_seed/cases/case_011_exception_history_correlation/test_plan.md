# Case 011: 异常历史相关性回放（超高难）

请生成一个 RV64 单线程 ISG 脚本，目标是在一次测试中触发多类异常并通过历史记录相关性进行阶段化验证。

## 任务目标

不仅要触发异常，还要让不同异常类别在时间顺序、来源特权级和返回路径上可对齐、可对比。

## 量化要求

1. 指令总数 130 到 220 条。
2. 异常事件不少于 5 次，至少覆盖 3 种异常类别。
3. 至少 2 次异常来自同一类别但不同触发点，用于历史相关性比对。
4. 至少 2 次异常来自不同源特权级上下文。
5. 至少 3 次异常后返回到主线并执行后继验证块。
6. CSR/状态相关指令不少于 24 条。
7. 结束前写 x10/a0 = 0，然后发出 M5EXIT##RISCV。

## 指令级结构要求

1. 阶段 A：异常基线与处理上下文准备。
2. 阶段 B：第一异常族触发（重复两次，触发点不同）。
3. 阶段 C：第二/第三异常族触发（跨特权上下文）。
4. 阶段 D：异常后关联验证（按类别与顺序进行标记对齐）。
5. 阶段 E：收敛退出。

## 数据编排要求

1. 维护 `TRAP_CLASS_SET` 与 `TRAP_SITE_SET` 两组集合，至少形成 5 个 (class, site) 样本。
2. 每次触发前写入阶段签名寄存器，触发后写入返回签名寄存器。
3. 至少两组同类异常使用不同触发点，验证历史记录顺序而非单点命中。
4. 在不同异常族之间插入短普通块，确保 trace 可分段。

## 通过判定

1. trace 可统计至少 5 次异常、3 种类别。
2. 同类异点异常可在时间顺序上区分并完成回放对齐。
3. 异常后主线恢复块可见，最终正常退出。

## 实现边界

- 仅限 RV64 单线程。
- 不引入向量主线。
- 重点是异常历史相关性，不是单一 fault 命中。

## 交付物

- 一个完整的 FORCE-RISCV Python 脚本。
- 一段简短说明，解释异常类别、触发点和返回链如何形成可验证相关性。
